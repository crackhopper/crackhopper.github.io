领域驱动设计简介

Mar 01, 2026
2 minute read

目前编写的软件比较上规模。偶尔看见DDD设计的书，发现和自己开发过程中的常见经验的总结非常一致。有的部分甚至更进了一步。因此，特意开一系列文章，记录和讲解这部分内容。

主要和AI Chat学习+选择了一本精简版本《Domain-Driven Design Quickly》 。

DDD 核心思想

领域驱动设计（Domain-Driven Design，简称 DDD）是一种以业务领域为核心的软件分析与设计方法论，最早由 Eric Evans 在《Domain-Driven Design: Tackling Complexity in the Heart of Software》一书中系统提出。DDD 并不是某种具体技术或框架，而是一套应对复杂业务系统的软件建模思想与实践体系。

在复杂业务系统中，常见问题包括：

  • 业务规则分散在代码各处，难以理解和维护
  • 技术模型主导设计，业务含义被弱化
  • 不同团队对同一概念理解不一致，沟通成本高
  • 系统规模扩大后，修改成本指数级上升

以领域为中心（Domain-Centric）

领域（Domain） 指的是软件要解决的业务问题空间，例如：

  • 电商系统中的「订单、支付、库存」
  • 金融系统中的「账户、清算、风控」

DDD 强调：设计应由业务领域驱动，而非数据库或技术框架驱动。

通用语言（Ubiquitous Language）

通用语言是 DDD 的基石之一：

  • 由开发人员与业务专家共同定义
  • 贯穿需求文档、会议讨论、设计模型、代码实现
  • 一个词在一个上下文中只能有一个明确含义

示例：

  • “订单”是已支付的，还是下单即生成？
  • “取消”是否允许已发货？

这些必须在模型中明确，而非隐含在代码逻辑中。

模型驱动设计（Model-Driven Design）

DDD 中的模型不仅是概念图，而是：

  • 可执行的
  • 与代码结构高度一致
  • 能直接反映业务规则

模型 = 业务知识的载体

战略设计（Strategic Design）

战略设计关注的是：系统如何拆分，边界如何划定，模型如何协作

子领域（Subdomain）

将整个业务领域拆分为多个子领域：

  • 核心子领域（Core Domain）
    企业核心竞争力，最值得投入资源
  • 支撑子领域（Supporting Domain）
    为核心领域服务，但非竞争优势
  • 通用子领域（Generic Domain）
    可复用或外购，如身份认证、日志

限界上下文（Bounded Context）

限界上下文： 一个模型在其中具有明确含义和一致解释的边界。

当多个团队开发一个项目时，代码开发是并行完成的，每个团队被分配了模型的一个特定部分。那些部分不是独立的，相互之间存在或多或少的关联。它们都是从一个大的模型出发，然后实现其中的一部分。假设某个团队创建了一个模块，然后提供给其他的团队使用。另一个团队的开发人员开始使用这个模块，但发现还缺少一些自己模块需要的功能，于是他添加了这个功能并提交了代码，以便所有的人都能使用。但是他可能没有意识到，这其实是对模型的一个变更，这个变更有可能会破坏应用的功能。这种情况很容易发生，因为没有人会花时间去完全理解整个模型。每个人都知道自己的后院里有什么，但对其他地方却并不是非常了解。

一个统一的企业模型是难以达到的理想状态，有的时候甚至都不值得去尝试。这些项目需要很多团队的通力协作。在开发过程中，团队需要高度的独立性，因为他们没有时间去经常开会和讨论设计。要协调这些团队是一项非常艰难的任务。也许他们属于不同的部门，有着独立的管理。当模型的设计在局部独立进化时，我们就面临着失去模型完整性的可能。努力为整个企业项目维护一个大的统一模型，以获得模型完整性，这个办法也不会有什么作用。解决方案并非显而易见，因为它与我们目前已经学到所有知识恰恰相反。不是试图保持一个迟早要四分五裂的大模型，我们应该做的是有意识地将大模型分解成多个较小的模型。只要遵守它们所绑定的契约，良好整合的小模型能够独立进化。每个模型都应该有一个清晰的边界，模型之间的关系也应该被精确地定义

每一个模型都有一个上下文。在我们处理一个独立的模型时，上下文是隐含的，我们不需要去定义它。当我们创建一个假定要和其他软件（例如一个遗留应用）交互的应用时，很明显新的应用有自己的模型和上下文，独立于遗留模型及其上下文。它们无法被合并、混合或者混淆起来。所以当我们开发大的企业应用时，我们需要为每一个我们创建的模型定义上下文。

如何将一个大的模型分解成小的模型没有什么固定的准则。尽量把那些相关联的以及能形成一个自然概念的元素放在一个模型里。模型应该足够小，以便能分配给一个团队去实现。团队协作和沟通会更加流畅，这会有助于开发人员共同完成一个模型。模型的上下文是一些条件的集合，这些条件可以确保应用在模型里的术语都有一个明确的含义。

明确定义模型所应用的上下文，根据以下因素来明确设置边界：

  • 团队的组织结构
  • 应用的特定部分中的惯例
  • 物理表现（例如代码库、数据库 Schema）

界定的上下文并不是模块。界定的上下文提供有模型在其中进化的逻辑框架。模块是被用来组织模型的元素，因此界定的上下文包含了模块。

举例来说：游戏本身、和游戏工具链。可以分别定义规范来做。

上下文映射（Context Mapping）

上下文映射用于回答一个关键问题：不同限界上下文之间，如何在保持各自模型纯洁性的前提下进行协作？它描述：

  • 上下文之间的协作方式
  • 模型的控制权归属
  • 变化的传播方向
  • 团队间的组织关系 上下文映射并不是代码层面的技术选型，而是架构与组织协同的契约。

防腐层（Anti-Corruption Layer，ACL）

防腐层是一层隔离机制，用于：防止外部系统或遗留系统的模型“腐蚀”本上下文的领域模型。

适用场景

  • 对接 遗留系统（Legacy System）
  • 集成第三方系统（ERP、支付、物流）
  • 不受你控制的上下文
  • 外部模型质量差、命名混乱、规则不清晰

实现方式

  • DTO 转换层
  • Adapter / Facade
  • 防腐 Service
  • 防腐 Repository
本领域模型
   ↑
防腐层（转换 / 适配）
   ↑
外部系统模型

常见误区

  • 把 ACL 当成“万能中转层”，堆业务逻辑
  • 在领域实体中直接依赖外部 DTO
  • 为内部系统之间随意使用 ACL（过度设计）

共享内核（Shared Kernel）

共享内核指的是：多个限界上下文共享一小部分模型与代码，并共同维护。 强调：

  • 共享范围必须 极小
  • 双方对变更高度协同

适用场景

  • 上下文关系紧密
  • 团队之间沟通顺畅
  • 共享部分变化频率低
  • 共享的是“稳定且通用”的核心概念

实现方式

  • 独立的 shared-kernel 模块 / 包
  • 明确版本与发布策略
  • 严格代码审查

常见误区

  • 共享范围过大
  • 把“公共工具类”误当共享内核
    • 区别在于： 是否承载“领域含义”
  • 无治理机制，导致“共享垃圾堆”

客户-供应商（Customer–Supplier）

该关系强调：一个上下文（客户）依赖另一个上下文（供应商），供应商需要对客户的需求负责。

适用场景

  • 明确的上下游关系
  • 上游系统由内部团队维护
  • 下游对上游有功能依赖
  • 示例：结算系统（客户）上游是 订单系统（供应商）

开放主机服务（Open Host Service，OHS）

开放主机服务是指：一个上下文以标准化、稳定的接口形式，对外提供服务。

适用场景

  • 被多个上下文或系统依赖
  • 需要长期稳定对外能力
  • 微服务 API 暴露

实现方式

  • REST / gRPC / GraphQL API
  • 明确 API 契约
  • 版本控制
  • 文档化
  • 通常与 发布语言（Published Language） 配合使用：使用统一、明确的 DTO 语言

战术设计（Tactical Design）

面向对象设计(Object Oriented Design, OOD): 把用例(UseCase)，用语言描述；对里面的名词(建模成Class)、动词(建模成Method)；用Class和Method的交互，完成对用例的实现 。

战术设计关注的是： 单个限界上下文内部如何建模和落地

实体（Entity）

  • 具有唯一标识
  • 生命周期中状态可变
  • 关注“是谁”，而非“值是什么”
  • 示例：订单、用户

OOP 语言会把对象的实例放于内存中，它们会为每个对象保持一个对象引用或者记录一个对象地址。在给定的某个时刻，这种引用对每一个对象而言是唯一的，但是很难保证在不确定的某个时间段内它也是如此。实际上恰恰相反。对象经常被移出或者移回内存，它们被序列化后在网络上传输，然后在另一端被重新创建，或者它们被销毁了。在程序的运行环境中，那个看起来像标识符的对象引用其实并不是我们正在谈论的标识符。如果有一个存放了天气信息（如温度）的类，很容易产生同一个类的不同实例，这两个实例都包含了同样的值，这两个对象是完全相当的，相互之间可以互换，但是它们拥有不同的引用，它们并不是实体。

在软件中实现实体意味着创建标识符。对一个人而言，其标识符可能是属性的组合：名称、出生日期、出生地、父母姓名、当前地址。系统能够轻易区分开两个拥有不同标识符的对象，或者将两个使用了相同标识符的对象看做是相同的。如果不能满足这个条件，整个系统可能会变得破烂不堪。

实体 是领域模型中非常重要的对象，并且它们应该在建模过程开始时就被考虑。决定一个对象是否需要成为一个实体也很重要。

值对象（Value Object）

  • 无唯一标识
  • 不可变
  • 通过属性值判断相等性
  • 示例：金额、地址、时间段

如果所有对象都建模成实体，那么会导致系统不必要的复杂。

考虑一个绘画应用。用户会看到一个画布且他能够用任何宽度、样式和颜色来画任何点和线。创建一个叫做 Point 的类非常有用，程序会对画布上的每一个点创建这个类的一个实例。这样的一个点会包含两个属性，对应于屏幕或者画布的坐标。是否有必要考虑每一个点都拥有一个标识符？这个标示符会有延续性吗？对于这样一个对象而言，看起来重要的事情只有它的坐标而已。

有的时候我们需要包含一个领域对象的某些属性。我们对它是哪一个对象并不感兴趣，而是只关心它所拥有的属性。用来描述领域的特定方面、并且没有标识符的一个对象，叫做值对象。

没有标识符，值对象就可以被轻易地创建或者丢弃。没有人关心创建一个标识符，在没有其他对象引用时，垃圾回收会处理这个对象。这极大简化了设计。

极力推荐将值对象实现为不可变的。它们由一个构造器创建，并且在它们的生命周期内永远不会被修改。当你想要得到这个对象的不同的值时，你简单地创建另一个对象就行了。

（实际上我们游戏系统的buff，更适合作为值对象；卡牌适合作为实体对象；当前的实现上都作为了实体，为了解决一些重复性buff叠加的问题，只是更简化的处理方式，这样做还是有一定缺陷的。）

一条箴言是： 如果值对象是可共享的，那么它们应该是不可变的。 值对象应该保持很小、很简单。当其他参与方需要一个值对象时，可以简单地传递值，或者创建一个副本。制作一个值对象的副本是非常简单的，通常不会有什么副作用。如果没有标识符，你可以按你所需创建若干副本，然后根据需要来销毁它们。

聚合与聚合根（Aggregate / Aggregate Root）

聚合是强一致性边界：

  • 聚合内部保证业务不变式
  • 外部只能通过聚合根访问
  • 示例：订单是聚合根，订单项不能被外部直接修改。

聚合是针对数据变化可以考虑成一个单元的一组关联的对象。

聚合使用边界将内部和外部的对象划分开来。每个聚合都有一个根。这个根是一个实体，并且它是外部可以访问的唯一的对象。根对象可以持有对任意聚合对象的引用，其他的对象可以互相持有彼此的引用，但一个外部对象只能持有对根对象的引用。如果边界内还有其他的实体，那些实体的标识符是本地化的，只在聚合内有意义。

如果一个聚合中的对象被保存到数据库中，可以通过查询来获得的应该只有根对象。其他的对象只能通过从根对象出发导航关联对象来获得。

工厂(Factory)

实体和聚合常常会很大很复杂，过于复杂以至于难以通过根实体的构造器来创建。实际上通过构造器努力构建一个复杂的聚合，并不是领域本身通常应该做的事情，在领域中，某些事物通常是由别的事物创建的（例如电器是在生产线上被创建的）。通过根实体的构造器来构建复杂的聚合，看上去就像是要用打印机构建打印机本身。

工厂被用来封装对象创建所必需的知识，它们对创建聚合特别有用。当聚合的根被创建后，所有聚合包含的对象将随之创建，所有的不变量得到了强化。

  • 工厂方法 是一个对象方法，包含并隐藏了必要的创建其他对象的知识。这在客户试图创建一个属于某聚合的对象时是非常有用的。解决方案是给聚合的根添加一个方法，由这个方法来负责对象的创建，强化所有的不变量，返回对那个对象的一个引用或者一个副本。
  • 工厂类：有时候创建一个对象的逻辑会更为复杂，或者创建一个对象还涉及到创建其他一系列对象时（例如，创建一个聚合）。可以使用一个单独的工厂对象来隐藏聚合的内部构造所需要的任务。
  • 抽象工厂 ： 当工厂内部有过多if/else用来创建 一族对象的时候。考虑抽象工厂接口，通过继承和实例化来简化逻辑，提供扩展性和可读性。

领域服务（Domain Service）

当业务逻辑，不适合放在某个实体或值对象中，又属于领域逻辑，则使用领域服务。

在我们开发通用语言时，领域中的关键概念被引入到语言中，语言中的名词很容易被映射成对象。语言中对应那些名词的动词成为了那些对象的行为。但是 有些领域中的动作，它们是一些动词，看上去却不属于任何对象 。它们代表了领域中的一个 重要的行为 ，所以不能忽略它们或者简单地把它们合并到某个实体或者值对象中。

例如，为了从一个账户向另一个账户转钱，这个功能应该放到转出的账户还是在接收的账户中？感觉放在这两个中的哪一个也不对劲。

当这样的行为从领域中被识别出来时，最佳实践是将它声明成一个服务。这样的对象不再拥有内置的状态，它的作用仅仅是为领域提供相应的功能。服务所能提供的帮助作用是非常重要的，一个服务可以将服务于实体和值对象的相关功能进行分组。

在 DDD 实践中，领域服务（Domain Service）与应用层（Application Layer）最容易被混淆，

  • 应用层解决的是「事情怎么发生」；
  • 领域服务解决的是「事情在业务上怎么算」。 本质区别，应用层关心用例（设计上更偏向用户侧，偏向于编排和调度领域对象)，领域服务（更关心规则，不关心谁用，由应用层来编排）

一个服务不应该替代通常隶属于领域对象的操作。我们不应该为每一个需要的操作创建一个服务。但是当一个操作凸现为领域中的一个重要概念时，就需要为它创建一个服务了。以下是服务的 3 个特征：

  • 服务执行的操作代表了一个领域概念，这个领域概念无法自然地隶属于一个实体或者值对象。
  • 被执行的操作涉及到领域中的其他的对象。
  • 操作是无状态的。

当使用服务时，保持领域层的隔离非常重要。很容易弄混属于领域层的服务和属于基础设施层的服务。应用层中也可能会有服务，这增加了额外的复杂性。将这些服务与领域层中相关服务分离开来会更加困难。当我们在设计阶段建立模型时，我们需要确保将领域层与其他的层隔离开来。（我通常使用XxxManager这样的格式；但不够好，更好的方式还是根据语义来取名，同时应用层强调Flow/UseCaseService，基础层强调技术术语Service）

模块(Module)

对一个大型的复杂项目而言，模型趋向于越来越大。当模型发展到了某个规模，将它作为整体来讨论很困难，理解不同部件之间的关系和交互变得很困难。基于这个原因，必须将模型组织到模块中。模块被用来作为组织相关概念和任务以便降低复杂性的一种方法。

推荐的做法是将高关联度的类分组到一个模块，以提供尽可能大的内聚性。有很多类型的内 聚性。最常用到的两个是

  • 通信性内聚（communicational cohesion）
    • 在模块中的部件操作相同的数据时，可以得到通信性内聚。把它们分到一组很有意义，因为它们之间存在很强的关联性。
  • 功能性内聚（functional cohesion）
    • 在模块中的部件协同工作以完成定义好的任务时，可以得到功能性内聚。功能性内聚被认为是最佳的内聚类型。

资源库(repos)

客户程序需要有一个实用的手段来获取已存在领域对象的引用。如果基础设施使得获得引用很容易，客户程序的开发人员可能会增加更多可导航的关联，导致模型混乱不堪。从另一方面讲，他们可能使用查询从数据库中获取所需的数据，或者拿到几个特定的对象，而不是通过聚合的根来导航。领域逻辑分散到查询和客户代码中，实体和值对象变得仅仅是数据的容器。应用了众多数据库访问基础设施，技术复杂性会迅速蔓延在客户代码中，开发人员降低了领域层的重要性，所做的工作跟模型无关了。总体的结果是丢失了对领域的专注，设计遭受了损害。

因此，使用一个资源库，它的目的是封装所有获取对象引用所需的逻辑。领域对象不需处理基础设施，以得到领域中对其他对象的引用。只需要从资源库中获取它们，于是模型重获它应有的清晰和专注。

资源库可以包含用来访问基础设施的细节信息，但它的接口应该非常简单。资源库应该拥有一组用来检索对象的方法。客户程序调用这样的方法，传递一个或者多个代表筛选条件的参数用来选择一个或者一组匹配的对象。可以通过传递实体的标识符来轻易指定一个实体。其他筛选条件可能由一组对象属性构成。资源库将所有的对象与这组条件来比较，并返回符合条件的那些对象。资源库接口可能还包含用来执行某些辅助计算（例如获取特定类型对象的数量）的方法。

需要注意的是，资源库的实现可能会非常像是基础设施，然而资源库的接口却是纯粹的领域模型。

领域事件（Domain Event）

用于表示领域中已发生的重要事实：

  • 通常是过去时
  • 用于解耦不同模块
  • 常用于最终一致性
  • 示例：OrderPaid, InventoryReserved

DDD 与分层架构

经典 DDD 分层包括：

  1. 用户接口层（UI / API）
  2. 应用层（Application Layer）
  3. 领域层（Domain Layer）
  4. 基础设施层（Infrastructure Layer）

其中：

  • 领域层是核心
  • 应用层负责协调，不承载业务规则

用户接口层

负责向用户展现信息以及解释用户命令。

对应我的游戏开发中，用户接口层主要是和引擎提供的UI组件通信，以及UI提供给用户访问的能力。

  • 不直接操作领域对象
  • 不包含用例流程
  • 只负责：
    • 输入 转化为 Command （应用层调用）
    • ViewModel / DTO（应用层消息） 转化为 输出

应用层

用来协调应用的活动。它不包含业务逻辑。它也不保留业务对象的状态，但它保留有应用任务的进度状态。（ 应用层代码可能很多，只不过它不承载领域知识；有时候会说应用层薄，并不是代码是少 ）

对应我的游戏开发中，应用层负责实现具体的use case。侧重关注use case中的流程编排、失败处理。

领域层

本层包含关于领域的信息。这是业务软件的核心所在。在这里保留业务对象的状态。对业务对象和它们状态的持久化被委托给了基础设施层

对应我的游戏开发中，领域层负责某个聚合性的逻辑规则，比如，举例子以buff计算为例子：

  • Buff值对象、效果特点、分类、如何影响数值。（跟配置几乎一一对应的一个值对象）
  • Buff结算实体（玩家/怪物身上的状态中的一个字段）：如何组合多个Buff实体合理结算。（聚合根）
  • 领域层之间通过“值对象 / 描述对象”交互，而不是互相依赖。
    • 卡牌聚合Buff：
    • Buff结算期聚合 卡牌组
  • Buff的作用域，提供描述化字段。比如：
    • Buff可以控制某个阶段的行为（比如抽牌）。那么Buff需要有字段描述作用阶段。
    • Buff应该：声明能力和约束
    • Buff不应该：控制流程执行（比如控制具体游戏阶段的切换，这样会造成耦合；目前我们设计上是允许这么做的，能力边界上存在一定的问题）

基础设施层

本层作为其他层的支撑库存在。它提供了层间的通信，实现对业务对象的持久化，包含对用户界面层的支持库等作用

  • Repository 实现
  • 配置加载
  • 引擎 API 适配
  • 事件总线
  • IO / 网络

实施 DDD 的关键原则

  • 从复杂核心业务入手，而非全系统一次性 DDD
  • 先做领域建模，再写代码
  • 保持模型与代码的一致性
  • 避免“形式主义 DDD”（只有名词，没有业务规则）
  • DDD ≠ 微服务，但高度契合微服务架构

适用与不适用场景

适用场景

  • 业务规则复杂、变化频繁
  • 多团队协作
  • 中大型系统

不适用场景

  • CRUD 为主的简单系统
  • 生命周期极短的项目
